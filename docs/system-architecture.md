# System Architecture

## Overview
Remote Claude Code controller — a full-stack web application to submit and monitor Claude Code tasks on an EC2 instance from any browser or iOS device.

```
User (iOS/Browser)
    ↓
Next.js Frontend (web app, Caddyfile-routed)
    ↓ HTTPS (via Caddy)
    ├→ Go API Server (job queue, subprocess runner, SSE streaming)
    ├→ SQLite (job history and results)
    └→ Claude Code CLI (subprocesses run on target repo)
```

## Deployment Architecture

### EC2 Instance Layout
```
/
├── /opt/atomiton/claudemote/        # app root (WORK_DIR for local dev, edit in system.cfg.json)
│   ├── backend/                      # Go source
│   │   ├── server                    # compiled binary
│   │   ├── cmd/
│   │   ├── internal/
│   │   └── .env                      # secrets only (gitignored)
│   ├── frontend/                     # Next.js source
│   │   ├── .next/                    # production build (recreated on deploy)
│   │   └── .env.local                # secrets only (gitignored)
│   ├── ecosystem.config.cjs          # pm2 process manager config
│   ├── system.cfg.json               # single source of truth for non-secret config
│   ├── Caddyfile.template            # reverse proxy template (rendered by start.sh)
│   ├── Caddyfile                     # generated by start.sh during Phase 2 (not in git)
│   └── start.sh                      # deploy/bootstrap script
│
├── /var/lib/claudemote/              # runtime state
│   └── claudemote.db                 # SQLite database
│
├── /etc/caddy/                       # Caddy config (systemd-managed)
│   └── Caddyfile                     # generated from template on bootstrap
│
└── /var/log/caddy/                   # Caddy logs (systemd journal also available)
```

### Process Architecture (pm2)
Two processes managed by pm2, started by `ecosystem.config.cjs`:

**1. claudemote-api** (Go HTTP server)
- Listens on `system.cfg.json:api.port` (default 8888)
- Implements `/api/*` endpoints (job submission, history, streaming)
- Spawns Claude Code subprocesses (count = `system.cfg.json:worker.count`)
- Writes job results to SQLite
- Reads non-secret config from `system.cfg.json` (injected as env by pm2)
- Reads secrets from `backend/.env` at startup

**2. claudemote-web** (Next.js)
- Listens on `system.cfg.json:web.port` (default 8088)
- Serves UI, handles `/api/auth/*` (NextAuth)
- Makes server-side API calls to `localhost:8888` (BACKEND_URL)
- Public client calls use same-origin via Caddy reverse proxy
- Reads secrets from `frontend/.env.local` at startup

### Caddy Reverse Proxy (systemd)
- Listens on :80 (HTTP) and :443 (HTTPS)
- Serves `system.cfg.json:hostname` (e.g. `claudemote.example.com`)
- Auto-provisions TLS cert from Let's Encrypt (requires DNS A record pointing to EC2)
- Routes:
  - `/api/jobs/*/stream` → Go backend with `flush_interval -1` (SSE passthrough)
  - `/api/auth/*` → Next.js (NextAuth routes)
  - `/api/*` → Go backend
  - Everything else → Next.js (pages, static assets)
- Compresses responses except SSE streams

## Configuration Management

### system.cfg.json (Committed, Non-Secret)
Single source of truth for all non-secret configuration. Read by:
- `start.sh` (via jq) — generates Caddyfile, runs health checks
- `ecosystem.config.cjs` — injects into env vars for pm2 processes

**Keys:**
```json
{
  "hostname": "claudemote.example.com",
  "api": { "port": 8888 },
  "web": { "port": 8088 },
  "worker": {
    "count": 2,
    "model": "claude-sonnet-4-6",
    "permission_mode": "bypassPermissions"
  },
  "jobs": {
    "timeout_min": 30,
    "max_cost_usd": 1.0,
    "log_retention_days": 14
  },
  "db_path": "/var/lib/claudemote/claudemote.db",
  "work_dir": "/opt/atomiton/claudemote",
  "pm2_max_memory": "500M"
}
```

### .env Files (Gitignored, Secrets Only)

**backend/.env:**
```
JWT_SECRET=<random 32+ hex>
ADMIN_USERNAME=admin
ADMIN_PASSWORD_HASH=<bcrypt hash>
CLAUDE_BIN=/path/to/claude
DB_PATH=/var/lib/claudemote/claudemote.db
```

**frontend/.env.local:**
```
AUTH_SECRET=<random 32+ base64>
NEXTAUTH_SECRET=<same as AUTH_SECRET>
NEXTAUTH_URL=https://claudemote.example.com
BACKEND_URL=http://localhost:8888
```

All other config (ports, hostname, worker settings, job limits, paths) lives in `system.cfg.json`.

## Data Flow

### Job Submission
1. User enters command in Next.js UI
2. Frontend calls `POST /api/auth/login` (NextAuth flow)
3. Frontend calls `POST /api/jobs` with JWT token
4. Go API creates Job record in SQLite, queues it
5. Response includes `job_id` immediately (async)

### Job Execution
1. Worker goroutine dequeues job
2. Runs: `claude -p "<command>" --output-format json` as subprocess
3. Reads result JSON (or captures stderr if failure)
4. Updates Job record with status, summary, timestamps
5. Closes job (marks complete or failed)

### Job Streaming
1. Frontend polls `GET /api/jobs/$id` every 2-3s (or calls streaming endpoint)
2. Go API returns job status + summary from SQLite
3. For real-time streaming: `GET /api/jobs/$id/stream` → SSE events
4. Caddy forwards SSE stream with `flush_interval -1` (no buffering)

## Security Model

**Trust model:** Admin password = root access to `work_dir`.
- Admin can submit any Claude Code command
- Claude Code runs with `bypassPermissions` mode
- Grants full shell access to `work_dir` filesystem
- Treat admin credentials like SSH keys (strong, rotated, not shared)

**Secrets isolation:**
- `.env` files gitignored, not in git
- `system.cfg.json` committed (non-secret only)
- Env vars injected at pm2 startup, not visible in process listings unless you're the owner
- NextAuth v5 session tokens (JWT) expire, validated server-side

**TLS:**
- Caddy auto-provisions Let's Encrypt cert (requires DNS + port 80/443 open)
- Cert renewal automatic, no manual renewal needed

## Performance Characteristics

- **Concurrent jobs:** Limited by `system.cfg.json:worker.count` (default 2)
- **Job timeout:** `system.cfg.json:jobs.timeout_min` (default 30 min wall-clock)
- **Database:** SQLite (single-file, ~100 jobs/second write throughput typical)
- **SSE latency:** Sub-second (flush_interval -1 disables buffering)
- **Memory:** pm2 configured to restart if exceeds `system.cfg.json:pm2_max_memory`

## Deployment Workflow

### First Deploy (./start.sh --bootstrap)
3-phase process:
1. **Phase 1 Discovery:** Detect OS, install tools, prompt for config, verify Claude auth
2. **Phase 2 Configure:** Generate env files, Caddyfile, create DB directory
3. **Phase 3 Build & Start:** Compile, migrate, start pm2, verify runtime

### Recurring Deploys (./start.sh)
```bash
./start.sh  # build backend + frontend, reload pm2
```

Changes to `system.cfg.json` are picked up by pm2 reload:
```bash
pm2 reload ecosystem.config.cjs
```

### Configuration Changes
To change a port or any non-secret setting:
1. Edit `system.cfg.json`
2. Run `./start.sh` (rebuilds + reloads)

Changes to secrets require editing the `.env` files and restarting.
